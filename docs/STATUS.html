Resumen ejecutivo

Dónde está JChat 3.0 hoy, en una vista.

JChat 3.0 tiene un producto funcional y pulido en las 3 plataformas. El sistema de menús (web) está completo: 21 plantillas de diseño, 40 paletas de color aplicables por negocio, responsive, y modales que heredan el tema. El sistema de chat de proximidad funciona en web, iOS y Android con sub-chats, presencia múltiple, y scroll robusto. La app móvil, el dashboard de negocio y el cliente web-por-QR están operativos.

Lo que bloquea el lanzamiento a producción con pagos reales son dos temas de seguridad (P0-2 y P0-3): los totales de pedido no se recalculan en el servidor, y las Edge Functions confían en IDs del cliente. Además, la red de seguridad de tipos de TypeScript está apagada (ignoreBuildErrors), lo que ya causó un bug oculto.

En una frase: el producto está listo para demos y pruebas reales; falta cerrar la capa de seguridad de dinero y algunas features de retención (no leídos, TTL de mensajes) antes de cobrar de verdad.

Mejoras recomendadas (investigación 2026)

Basado en mejores prácticas actuales de la industria. Priorizadas.

⚡ Realtime a escala (Supabase)

AltaVigilar fugas de canales

Los canales sin cerrar son la causa #1 de topar límites de conexión. Ya cuidamos removeChannel en el port móvil — mantenerlo como regla en todo canal nuevo (especialmente con presencia múltiple = 3 canales por usuario).

Fuente: Supabase Realtime in Production (2026)

MediaPresencia obsoleta tras cambio de visibilidad

Presence se pone "stale" tras cambios de visibilidad de pestaña/app. El AppState que agregamos en móvil ataca esto; conviene una defensa equivalente en web (re-sync al volver a la pestaña).

Fuente: Supabase Realtime in Production (2026)

FuturoBroadcast-from-Database si crece la escala

Hoy el chat usa postgres_changes (~50-200ms), correcto para persistencia. Si superan ~500 conexiones concurrentes o el CPU de la BD sube, migrar a "Broadcast from Database" (un trigger difunde a un canal en vez de que cada cliente escuche la tabla). No urgente ahora.

Fuente: Supabase docs / TechLead 2026

💳 Seguridad de pagos (Stripe)

AltaIdempotency keys en todo POST

Añadir un idempotency key (UUID o customerID+orderID) a cada request de pago, para reintentar sin cobrar doble. Directamente aplicable al implementar el checkout.

Fuente: Stripe Server-side integration (2026)

AltaWebhooks como fuente de verdad, no el cliente

Nunca confiar solo en la confirmación del cliente — verificar el estado del pago vía webhook (con verificación de firma). Hacer los handlers idempotentes (guardar event.id, saltar duplicados). Esto refuerza P0-2/P0-3.

Fuente: Stripe Integration Guide 2026

MediaApple Pay / Google Pay para conversión

Los wallets suben conversión y son de los métodos más seguros (el número de tarjeta nunca sale del dispositivo). Buen encaje para el flujo móvil de pedido-en-mesa.

Fuente: Stripe payments security (2026)

🏗️ Arquitectura & proceso

AltaQuitar ignoreBuildErrors ya

La comunidad Supabase/Next 2026 es enfática: mantener el schema y los tipos en control de versiones y con el type-check activo evita el incidente clásico "funcionó en dev, cayó en prod". JChat ya vivió el riesgo (tipos corruptos ocultos).

Fuente: Supabase Production Architecture 2026

MediaPooler en modo transacción para Vercel

Como el web corre en Vercel (function-per-request), usar la URL del pooler en modo transacción y desactivar prepared statements — previene el incidente de conexiones más común de Supabase en producción.

Fuente: Supabase Production Architecture 2026
